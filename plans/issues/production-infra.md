# Production / Infrastructure Issues

Active production hardening and infrastructure concerns.

## [PROD-002] Game server DO — high "client disconnected" error rate (~56%)

**Priority**: Medium

101 errors out of 180 requests, nearly all "Client disconnected". Zero CPU limit exceeded.

**Likely cause**: WebSocket lifecycle — backgrounding app, losing mobile connectivity, browser tab suspension. Expected for mobile WebSocket game, but 56% suggests aggressive reconnection or flaky connections.

**Investigation needed**:
- Check `useGameEngine` reconnection logic for backoff
- Check server logs for repeated connect/disconnect from same player
- Consider exponential backoff for reconnection
- Consider heartbeat/ping for stale connection detection

**Status**: Investigating

## [PROD-003] Storage operations volume (6k in 24h)

**Priority**: Low

Each L2 context mutation triggers snapshot `ctx.storage.put()`. Duplicate SYNC issue doubles writes. Full snapshot on every save (no delta).

**Potential improvements**: Debounce writes (~500ms), separate volatile/durable state, eliminate duplicate subscription fire.

**Status**: Monitoring — not a cost concern yet, scales linearly with activity

## [PROD-004] Lobby -> Game Server communication uses public fetch

**Priority**: Medium

Lobby server actions use `fetch()` over public internet to game server DO. Adds latency, counts as subrequest, requires AUTH_SECRET.

**Fix**: Cloudflare service bindings — zero network cost, no auth overhead, type-safe RPC. Both workers in same account. Define game server as service binding in lobby's wrangler.json, expose RPC methods via `WorkerEntrypoint`, replace fetch() with direct invocations.

**Prerequisite**: PROD-006 fixed (ADR-070) — RPC methods now available.

**Status**: Planned improvement

## [PROD-011] D1 writes in subscription are unawaited (race condition window)

**Priority**: Low

D1 operations (updateGameEnd, creditGold, persistFactToD1) are fire-and-forget. Non-storage I/O opens the input gate, allowing request interleaving.

**Partial fix (ADR-070)**: `goldCredited` persisted to `ctx.storage.put()` and restored in `onStart()`. Prevents double-credit across restarts. D1 write race window remains but is acceptably small.

**Status**: Partially mitigated — goldCredited survives restarts; D1 write race window remains but acceptable

## [PROD-023] L1 subscription callback is a monolithic observer

**Priority**: Medium

Handles too many concerns: snapshot persistence, SYNC broadcast, tickers, gate-change tickers, D1 writes, gold payouts, task cleanup. Each concern needs tracking variables + `isRestoreFire` flag.

**Symptom**: Duplicate "DMs are now open!" and "Group chat is now open!" ticker messages (6-8x per session) — on DO wake, `actor.start()` fires subscription before tracking vars initialized.

**Proposed fix**: Extract ticker generation into a dedicated L1-level actor that owns its state (phase, gates, buffer) and naturally ignores restore.

**Status**: Band-aid fix shipped (`isRestoreFire` flag). Architectural refactor deferred.

## [PROD-026] Returning players see PWA install prompt and stale game shell

**Priority**: Medium

**Issue A**: Invite link opens Safari (not installed PWA). `PwaGate` shows install prompt for already-installed app.
**Short-term fix**: Skip install gate for returning users (check for existing `po_token_*` in localStorage).

**Issue B**: PWA launches into deleted/stale game. Dynamic `start_url` embeds game-specific JWT. If game deleted, shell renders empty DO.
**Root cause**: No game validity check between token decode and shell render.

**Long-term**: Push-based invites for returning players (push notification -> PWA opens -> join flow inside PWA). Deferred to post-playtest.

**Status**: Documented — short-term fix planned

## [PROD-028] Game code entry flow lacks validation

**Priority**: Medium

1. **Expired game code -> empty character selection**: Lobby issues session and redirects without verifying game exists
2. **Non-existent codes show "Game has ended"**: Should distinguish "doesn't exist" vs "ended"
3. **No centralized game validation**: Checks scattered across actions with inconsistent behavior
4. **Public vs private game tension**: Launcher code entry is an unauthenticated backdoor

**Proposed fix**: Shared validation middleware for every `/[code]` route: game exists? status? user is member? joinable?

**Status**: Documented — needs design decision on public vs private game model

## [PROD-030] Speed run mode creates false positives in local testing

**Priority**: High

Speed run compresses multi-day games into minutes, bypassing real-world flows. Directly caused a critical token purge bug (88571ff) going undetected until real-world playtest.

**Gaps**: Game lifecycle states (RECRUITING -> STARTED), token persistence timing, timeline event spacing, multi-device/PWA flows.

**Proposed fix**: Local testing protocol for real-world flows, "slow test" mode, E2E Playwright tests for recruiting/waiting state, pre-playtest checklist.

**Status**: Documented — needs testing strategy improvement
